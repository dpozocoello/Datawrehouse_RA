1. Flujos ETL/ELT y Dependencias (Sección 5)

El orquestador principal es run_full_rgd_etl.py. El flujo de trabajo se divide en 3 pasos críticos:

  1. Python Ingestion (Source to STG): Ejecuta ingesta_waste_chemical.py. Extrae datos crudos de las vistas del sistema transaccional (SUIA/COA). Se añadieron JOINs a public.provinces y public.cantons en la fuente para traer nombres reales.
  2. Geo-Resolution (Step 1.5): Si existen coordenadas pero no hay provincia/cantón, el script geo_resolver.py utiliza un archivo GeoJSON de fronteras administrativas para hacer un Reverse Geocoding (Point-in-Polygon) en memoria y actualiza la tabla de staging.
  3. SQL Transformation (STG to DW): Llama al sp_etl_waste_chemical(). Traslada los datos al Data Warehouse aplicando deduplicación y reglas de negocio.

2. Reglas de Negocio y Transformaciones (Sección 6)

  • Deduplicación DISTINCT ON: El SP utiliza SELECT DISTINCT ON sobre las llaves primarias. Es obligatorio que el bloque ORDER BY coincida posicionalmente con las llaves para evitar el error de base de datos InvalidColumnReference.
  • Casting Explícito: Todas las métricas numéricas en staging (inferidas como TEXT o FLOAT por Pandas) son forzadas a ::numeric o ::bigint antes de la inserción.
  • Manejo de Nulos (Fallback): Cualquier llave foránea que venga nula o no cruce, se asigna al valor por defecto 0 (Dimensión "N/A" o "Desconocido").

3. Calidad, Auditoría y Trazabilidad (Sección 7)

El sistema cuenta con una tabla de auditoría dw.etl_process_log que registra:

  • Nombre del proceso y modo de ejecución.
  • Tiempo de inicio, fin y duración.
  • Estado (SUCCESS/FAILED) y el error_message exacto capturado por el orquestador.

Adicionalmente, se recomienda correr el script check_gen_points.py semanalmente para identificar puntos huérfanos geográficamente que necesiten calibración manual en la fuente.
